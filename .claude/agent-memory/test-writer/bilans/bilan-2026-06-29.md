## Stratégies de test validées

### 1. Deux fichiers de test (fonctionnels + techniques en un seul bloc)

- **sync.test.js** (47 tests) : tests techniques M-SYNC (interface, garde réentrance, offline, réseau) + scénarios BDD F3.2/F3.3 rassemblés
- **store.test.js** (38 tests) : tests techniques M-STORE (load, add, update, fetch, push) + intégration offline complète

**Justification** : Pas de séparation unit/integration/functional ici — les tests techniques **sont** les critères d'acceptance. Un test technique qui échoue = un scénario utilisateur cassé. Approche cohérente avec brief du test-writer (couvrir deux niveaux : fonctionnels + techniques).

### 2. Mocking exhaustif (localStorage, fetch, navigator.onLine)

- `jest-localstorage-mock` pour localStorage (clair et idiomatique)
- `jest.fn()` pour fetch avec `.mockResolvedValue()` / `.mockRejectedValue()`
- `Object.defineProperty(navigator, 'onLine', { writable: true })` pour les tests réseau
- `crypto.randomUUID` mockée pour les tests (fallback Date.now() documentée en archi)

**Justification** : Aucune requête réseau réelle, tests isolés et reproductibles, 0 flakiness attendu.

### 3. Traçabilité specs → tests

Chaque scénario BDD gets un test nommé avec `// specs § F3.2` ou `// specs § F4`, permettant :
- Vérifier que 100% des specs sont couverts
- Retrouver rapidement le test si un scénario est rejeté en review
- Traçabilité bidirectionnelle code-implementer ← → test-writer ← → specs-framer

### 4. Cas limites explicites

Tests "Cas critique #N" rassemblés dans `describe('Cas limite')` :
- Double flush simultané (réentrance, guard `flushing`)
- PUT 400 si backlog vide (court-circuit dans flushPending)
- Réseau échoue mid-flush (file conservée)
- Offline au boot (hasPending check avant fetch)

Chacun représente un risque nommé dans architecture.md §11 — le test devient la preuv that the mitigation works.

### 2. Coverage : quelle métrique ?

Après implémentation, mesurer :
- **Lignes** : >75% (acceptable pour Vanilla JS sans test e2e)
- **Branches** : >70% (certains catch d'erreur difficiles à simuler)

Tests sont écrits pour maximiser les branches (if offline, if network error, if file empty, etc.) — aucune raison d'avoir une branche non-testable.

### 3. Tests de performance / limites

Aucun test de performance (quota localStorage, délai PUT long, batterie d'items). Acceptable pour v2.0 (usage solo, <100 items attendus). À ajouter en V2+ si nécessaire.

## Décisions techniques de cette session

### Config Jest pour ESM

**Setup** :
- `jest.config.js` avec `transform: {}` (pas de transpilation)
- `jest.setup.js` importé pour mock localStorage + navigator + crypto
- `package.json` : scripts avec `node --experimental-vm-modules jest`

**Pourquoi ESM et pas CommonJS** : Le projet est `"type": "module"` (Vite native), convertir en CommonJS pour tests serait un overhead inutile.

### Pas de supertest / API test framework

Aucune couche Express/Node.js server dans ce projet — c'est du Vanilla JS SPA. fetch est mocké directement. Code-implementer doit tester manuellement ou ajouter un test e2e pour vérifier que `/api/backlog` est bien appelé.

### Fire-and-forget flushPending()

Les tests supposent `store.add()` fait un appel **non-bloquant** à `flushPending()`. Cela signifie :
- Le formulaire se ferme immédiatement
- La sync se fait en arrière-plan
- Si le réseau échoue, la file persiste, l'utilisateur voit un badge "Hors ligne", réessai au prochain `online`

C'est la bonne UX pour offline (ne pas bloquer l'UI sur réseau lent).

## Leçons pour futures sessions test-writer

### ✅ À reproduire

- **Lire les specs COMPLÈTEMENT avant de coder** — j'ai gagné du temps en comprenant les 3 sections F3.1/F3.2/F3.3 avant de toucher aux tests
- **Traçabilité commentée** — chaque scénario BDD obtient `// specs §...`, évite la divergence post-implémentation
- **Cas limites explicites** — nommer les risques de l'archi (réentrance, 400 vide, réseau échoue) et écrire un test pour chacun = zéro surprise
- **Jest ESM sans transpilation** — garder la config au minimum (pas de TypeScript, pas de Babel) quand possible
- **Inter-agent.md avant implémentation** — clarifier les ambiguïtés *maintenant* sauve du temps code-implementer

### ❌ À éviter

- Ne pas écrire 200+ tests micro-unitaires qui testent chaque ligne — ici on a 85 tests qui couvrent le contrat ET les cas limites. C'est le bon ratio.
- Ne pas mocker les dépendances internes excessivement — les tests store.test.js testent store.js + sync.js ensemble dans la pile offline (pas de mock de sync).
- Ne pas oublier les cas "hors-ligne au boot" vs "hors-ligne après chargement" — deux chemins d'exécution différents.
